# Production Database Restore Guide

This guide explains how to restore the PostgreSQL database from the daily encrypted backups generated by the backup system (`infra/backup/backup.sh`).

---

## 1. Retrieve the Encrypted Backup File

Find the backup file you want to restore. It will be named like `dana_backup_YYYYMMDD_HHMMSS.sql.gz.enc`.

### For Local Backups:
The backup is stored on the dedicated host:
```bash
/var/backups/dana/dana_backup_20260530_120000.sql.gz.enc
```

### For S3-Compatible Storage:
Download the backup from your bucket (MinIO, R2, B2, Wasabi, etc.) using your preferred tool:
```bash
# E.g., using aws cli
aws s3 cp s3://my-bucket/dana_backup_20260530_120000.sql.gz.enc ./
```

---

## 2. Decrypt the Backup

To decrypt the backup, you must have the original `BACKUP_ENCRYPTION_KEY` used when the backup was created.

Run the following command to decrypt the file:
```bash
openssl enc -d -aes-256-cbc -pbkdf2 -pass pass:"YOUR_BACKUP_ENCRYPTION_KEY" \
  -in dana_backup_20260530_120000.sql.gz.enc \
  -out decrypted_backup.sql.gz
```

Verify that the decrypted file is a valid gzip file:
```bash
gzip -t decrypted_backup.sql.gz
```

---

## 3. Restore to PostgreSQL

### Step 3a: Decompress the File
Decompress the decrypted backup file:
```bash
gunzip decrypted_backup.sql.gz
# This will extract 'decrypted_backup.sql'
```

### Step 3b: Clean/Prepare the Database
If you are restoring over an existing database, it is best to drop the database and recreate it to avoid conflicts:
```bash
# Connect directly to postgres instance (bypass PgBouncer)
psql "postgresql://dana_user:password@postgres:5432/postgres" -c "DROP DATABASE dana;"
psql "postgresql://dana_user:password@postgres:5432/postgres" -c "CREATE DATABASE dana OWNER dana_user;"
```

### Step 3c: Import Schema and Data
Restore the schema and data:
```bash
psql "postgresql://dana_user:password@postgres:5432/dana" -f decrypted_backup.sql
```

Verify that the tables were restored correctly:
```bash
psql "postgresql://dana_user:password@postgres:5432/dana" -c "\dt"
```

---

## 4. Troubleshooting

*   **Error: Bad Magic Number**: This means the `BACKUP_ENCRYPTION_KEY` you provided during decryption does not match the key used during encryption.
*   **PgBouncer Connection Errors**: Do **not** run restores through PgBouncer on port `6432` if PgBouncer is in transaction pooling mode. Transaction pooling does not support DDL commands or multiple active sessions for data definition. Always restore directly to PostgreSQL (default port `5432`).
*   **Duplicate Key Violations**: Ensure you drop and recreate the database schema before restoring, or run the restore to a clean database instance.
